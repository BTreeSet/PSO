# AGENTS.md

Guidance for agents working on Proton-Singbox Orchestrator (PSO).

## Project Intent

PSO is a Rust control plane for automatic `sing-box` orchestration from ProtonVPN account state and declarative configuration. The intended production path is a compiled binary or container running `pso run` with a mounted state directory. Manual subcommands exist for setup and troubleshooting.

Keep three runtime concepts separate:

- App binary: `pso` and, in containers, the bundled `sing-box` binary.
- Declarative configuration: operator-authored files such as `pso.config.json` and `config.template.json`.
- State directory: user-opaque persistent runtime data managed by PSO, such as per-account VPN session state and Proton topology state.

## Architecture Rules

- Prefer config-file defaults plus small CLI overrides. Do not add one-off flags when the setting belongs in `pso.config.json`.
- Treat state as integral runtime data, not as an optional cache. If an operator-supplied fallback file exists, use it only when live API data and state cannot satisfy the operation as designed.
- Keep WireGuard private keys local. They must be generated by PSO and must not be accepted from the user or sent to Proton.
- Keep `sing-box` deployment SIGHUP-only. Validate config before replacing active config.
- Prefer explicit `sing-box` targeting by PID or executable path. Name-based process lookup is only a last fallback.
- Auth flows must support headless operation: password file/env, TOTP secret/env, mounted state directory, and CAPTCHA token retry.
- Health and recovery should be automatic in the `run` path. Manual commands should remain useful for troubleshooting but should not be the primary operational model.
- Multi-account work must keep each Proton account isolated in state and refresh WireGuard identity independently.

## Code Style

- Keep code small, typed, and composable. Push resolution logic into focused helpers rather than scattering defaults through command handlers.
- Prefer total-ish functions returning `Result<T>` with context over panics, `expect`, or production `unwrap`.
- Use iterator-style transformations where they improve clarity, but do not obscure error handling.
- Avoid historical compatibility shims. PSO is pre-alpha; choose the clean current interface.
- Keep dependencies current, feature-minimal, and documented in `docs/dependencies.md`.
- Do not log secrets: passwords, refresh tokens, access tokens, WireGuard private keys, or certificate bodies.

## Quality Gates

Run these before considering work complete:

```bash
cargo fmt --check
cargo clippy --locked --all-targets -- -D warnings
cargo test --locked
```

Also run targeted CLI help checks when editing the interface:

```bash
cargo run -- --help
cargo run -- run --help
cargo run -- auth --help
cargo run -- topology --help
cargo run -- control-plane --help
```

Docker is not available in every development environment. If local Docker is unavailable, state that container builds were not run and rely on the GitHub Actions Buildx workflow for image validation.

## Documentation Rules

- README should describe public, factual behavior only. Avoid internal planning language and avoid claiming features that are only on the roadmap.
- Put future ideas in `docs/roadmap.md`.
- Update `pso.config.example.json` when adding config-file fields.
- Update `docs/dependencies.md` when adding, removing, or materially changing dependencies/features.
